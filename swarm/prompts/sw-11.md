# SW-11: Optimise TUI startup and refresh performance

## Problem
`status --watch` is slow to load because:
1. `discoverProjects()` walks the entire directory tree looking for `swarm.toml` files
2. Every 3s refresh calls `tmux capture-pane` for EACH running agent (N subprocesses)
3. All tickets (including done) are iterated and rendered every cycle

## Scope
Optimise the TUI for fast startup and smooth refresh.

### Startup optimisation
1. `discoverProjects()` — limit `WalkDir` depth to 3 levels (no deep recursion)
2. Skip known heavy dirs: `node_modules`, `.next`, `.-worktrees`, `.git`, `.decapod`, `vendor`
3. Cache discovered projects for the session (don't re-walk on refresh)

### Refresh optimisation
1. Only call `tmux capture-pane` for running tickets (already done) but batch them:
   - Use a single goroutine pool (max 3 concurrent) instead of sequential calls
2. Skip progress parsing for done/todo/failed/blocked tickets (already done, but verify)
3. Only re-read tracker.json if mtime changed (stat check before read)
4. Debounce rapid refreshes

### Display optimisation
1. Add `--hide-done` flag to TUI (default: show all, but user can toggle with 'd' key)
2. When done tickets are hidden, show summary line: "12 done (hidden, press 'd' to show)"

### Files
- `internal/tui/tui.go` — optimise discoverProjects, refresh, add hide-done toggle
- `internal/tui/tui_test.go` — tests for discovery depth limit, mtime cache

## Additional fix: Column alignment in plain `status` output

In `cmd/status.go`, the plain `status` command uses `tabwriter` which over-pads when columns (especially DEPENDS) are empty. Fix: switch from tabwriter to fixed-width `fmt.Sprintf` formatting so columns stay consistent regardless of content:

```
fmt.Sprintf("%-8s %-6s %-8s %-18s %-12s %s", id, phase, status, branch, depends, desc)
```

This matches the cleaner MC-style output where columns don't shift based on content width.
